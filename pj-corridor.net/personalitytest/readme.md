# そんな時どうする生成 AI 連携アプリ開発

こんにちは、エンジニアの中山です。この記事では趣味プログラミングでのパーソナリティ診断アプリ開発を通じて直面した課題や気付き、例えばアビューズ対策やプロンプトチューニングにおけるトレードオフへの向き合い方についてご紹介させていただきます。なお、以前シナジーマーケティングでご一緒させて頂いたこともあり、TECHSCORE BLOG への記事掲載についてご快諾いただきました ^^ どうもありがとうございます。

最初にアプリのご紹介です。煩わしい広告（笑）はウインドウ幅を調整すれば消せます。

- <a href='https://pj-corridor.net/personalitytest/OpenCAPS.html'>CAPS（= Controller, Analyzer, Promoter, Supporter）診断</a>
- <a href='https://pj-corridor.net/personalitytest/OpenDiSC.html'>DiSC（= Drive, Influence, Steadiness, Compliance）診断</a>

CAPS や DiSC は MBTI と同様、いわゆる疑似科学的なパーソナリティ診断です。意思決定の根拠にはできませんが、有益な活用方法もあります。例えばプロジェクトの立ち上げ時にメンバーの診断結果を共有することで会話のきっかけになりますし、チームビルディングに役立ちます。

ちなみにこちらは私の診断結果です。

★（スクリーンキャプチャ）★

診断結果は生成 AI の出力を含んでいますが、品質担保にかけるコストは趣味プログラミングの範囲に抑える必要がありました。そのためルールベースの採点ロジックは deterministic に保ち、生成 AI は「説明文の生成」に制限することでハルシネーションの影響範囲をコントロールしています。

それでは、ここからは開発を通じて得た気付きを 3 つの章に分けてご紹介します。

## 1. アビューズ対策編

診断アプリは匿名でアクセスできますが、バックエンドには課金の発生する生成 AI（Amazon Bedrock）を利用しています。そのため、ボットによるアビューズで課金が膨らむ、いわゆる「課金爆発」は十分に起こり得るシナリオです。とはいえ、オーバースペックな対策は避けつつ攻撃側のインセンティブも考慮し「高度なアビューズは受容し、それ以外は遮断」の方針を定め、以下の観点で対策を検討しました。

- WAF による一般的な攻撃トラフィックの遮断
- 雑なボットや診断フローを経由しないブラウザのアクセス制御

前者の WAF については、AWS が標準的に提供している保護パックをベースに、今回の診断アプリには不要な対策（例えば SQL インジェクション）を削除し、固有の対策（例えば POST 回数の制限）を追加しました。ところで、保護パックに限らず AWS コンソールから設定した内容は忘れてしまうことも多く、暫定的な設定がトラブルにつながることもあるので

- 保護パックの <a href='https://github.com/nakayama-kazuki/202x/blob/main/.github/workflows/WAFPolicyApiCorridor.json'>JSON コード</a> を Source of Truth としてリポジトリで管理
- 保護パックの保守は、最初に Source of Truth を修正した後に AWS コンソールで JSON をペーストしエラーチェック
- CI で Source of Truth と稼働中の保護パックの <a href='https://github.com/nakayama-kazuki/202x/blob/main/.github/workflows/deploy-corridor.yml'>整合性チェック</a>

のように、インフラ設定もアプリケーションコードと同様に管理することにしました。

ここで一つ懸念事項がありました。今回は Lambda Function URL を公開エンドポイントとして利用していますが、CloudFront を経由しないリクエストが WAF をバイパスできてしまうことです。幾つか対策を考えましたが

- Lambda Function URL の <a href='https://github.com/nakayama-kazuki/202x/blob/main/.github/workflows/spot-private-lambda.yml'>private 化</a> は今回の構成では採用が難しい
- CloudFront で拡張ヘッダに秘密情報を付与し、それを Lambda 側で検証することで CloudFront 経由（つまりは WAF も経由）であることを担保する形はアプリケーションの環境依存を強めてしまう

という理由からいずれの対応も見送りとし、Lambda Function URL への直接アクセスは冒頭の方針に基づいていったん受容することにしました。

後者のアクセス制御については、

- 診断アプリアクセス時に token を Set-Cookie 
  - token は Lambda の環境変数に格納した秘密の情報 + 時刻情報から生成
- 診断クエリ経由で token を送信し、それを検証
  - 人間が設問回答に要するであろう時間の範囲を正常なクエリと判断するために時刻情報を利用

のような制御を実装しました。これとあわせて別レイヤーの対策 …

- SameSite=Strict（<a href='https://blog.techscore.com/entry/2023/10/06/110100'>参考</a>）として別ドメインからの POST クエリを遮断
- 許可リストにある origin のみを Access-Control-Allow-Origin に指定

も併用しました。ボットを高度化すれば上述の対策を迂回するアビューズは可能ですが、いったんはこの状態で運用～モニタリングしつつ、トラフィックに応じて追加の対策を検討することにします。

ところで、アビューズ対策には後日談があります。運用してみてわかったことですが、コストの観点では WAF が Bedrock よりもだいぶ大きくなってしまいました。マネージドサービスが常に最適解ではないことの一例かと思います。なので次のステップとしては WAF の利用を中止し、POST 回数の制限やキャッシュを DynamoDB で実装することを検討しています。

## 2. アプリケーション開発編

どうせなら世界中の人に試してもらえると楽しいですよね。なので、今回の診断アプリは完成後に Reddit に投稿することをゴールに考えていました。さらに、英語よりもブラウザ言語を用いた方がユーザー体験がよいだろうと <a href='https://github.com/nakayama-kazuki/202x/blob/main/pj-corridor.net/personalitytest/OpenAssessmentLib.js#L18'>i18n の仕組を実装し</a> 9 カ国語に対応しつつ、加えて生成 AI の出力が安定するように、

- ユーザーへの質問は母国語
  - 事前に翻訳済みのリソース
- 診断クエリとして生成 AI に入力するテキストは常に英語
- ユーザーへのフィードバックは母国語
  - 事前に翻訳済みのリソース + 生成 AI 出力

のような I/F 設計としました。

★（スクリーンキャプチャ）★

また、状態や UI 管理および Lambda との I/F の実装を DiSC と CAPS で共通化し、将来 MBTI などの診断アプリを開発する場合にも同じフレームワークを活用できるようにしました。

さて、ここで生成 AI 連携特有の懸念事項があります。それはプロンプトチューニングは想像以上に手間がかかることです。急がば回れで、繰り返しプロンプトのチューニングが発生することを前提とした Try & Error 基盤を用意しましょう。

- テスト環境（AI 接続なし）にて、テンプレートに質問回答を合成した結果を確認できる仕組みを用意する
  - それを Gemini や ChatGPT に投げて及第点の結果が得られるまでチューニングを行う
- プロンプト生成のためのショートカットを用意する
  - 診断アプリの場合は設問回答に時間がかかるのでランダムな自動回答のロジックを用意
    - テスト用のクエリパラメータを <a href='https://github.com/nakayama-kazuki/202x/blob/main/pj-corridor.net/personalitytest/OpenCAPS.html#L2054'>SHA256 で検証</a> することでアビューズリスク回避

このような仕組みを活用してプロンプトをレビュー（これも Gemini や ChatGPT を利用）しつつ構造のブラッシュアップを進めていきます。プロンプト自体が時間とともに長くなりがちなので、随時指示の統廃合や構造の見直しといったリファクタリングも重要です。

## 3. 技術選定編

まず悩んだのが Lambda の方式です。

- Lambda zip 方式（Python 利用）
- Lambda 独自コンテナ方式（PHP 利用）

当初、ローカルに PHP のテスト環境を構築済みだったので、すぐにでもアプリケーション開発～テストに着手できそうな独自コンテナ方式に目が行きましたが、Lambda との親和性や CI の複雑化の懸念があり見送ることにしました。そして、後で楽をするために <a href='https://github.com/nakayama-kazuki/202x/blob/main/testenv/scripts/template.py'>ローカル / Lambda 共通テンプレート</a> や環境変数をケアする <a href='https://github.com/nakayama-kazuki/202x/blob/main/testenv/scripts/restart-python.bat'>Python ランチャー</a> を用意しました。この仕組みが前述のプロンプトチューニングの Try & Error 基盤につながっています。

蛇足ですが AWS 環境で試行錯誤をすると Lambda が勝手に Role を作るので、忘れないうちに削除しておきましょう。

次いで悩んだのは API Gateway を利用するか否かについてです。

- CloudFront → Lambda → Bedrock
- CloudFront → API Gateway → Lambda → Bedrock

API Gateway が提供する機能毎に

- レート制御 → WAF で代替
- 認証 → アプリケーションのアクセス制御で代替
- ルーティング → アプリケーション側で代替

としているため、今回は導入を見送り将来機能拡張することがあれば再検討することにします。
