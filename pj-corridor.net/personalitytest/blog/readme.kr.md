# 생성형 AI 연계 애플리케이션 개발: 실제 트레이드오프에서 얻은 교훈

안녕하세요, 엔지니어 나카야마입니다. 이 글에서는 생성형 AI를 활용한 애플리케이션을 개발하면서 직면했던 과제들—특히 어뷰즈 대응과 프롬프트 튜닝—그리고 그 과정에서 마주한 트레이드오프에 대해 소개합니다. 과거 시너지 마케팅과 함께 일한 인연으로 TECHSCORE BLOG에 글을 게재할 기회를 얻게 되었습니다 😊

먼저 아래의 성격 진단 애플리케이션을 직접 체험해 보세요:

- CAPS (Controller, Analyzer, Promoter, Supporter)  
  https://pj-corridor.net/personalitytest/OpenCAPS.html  
- DiSC (Drive, Influence, Steadiness, Compliance)  
  https://pj-corridor.net/personalitytest/OpenDiSC.html  

CAPS와 DiSC는 MBTI와 마찬가지로 일반적으로 의사과학적이라고 여겨지는 성격 진단으로 분류됩니다. 의사결정의 근거로 사용하기에는 적절하지 않지만, 워크숍에서 참가자들이 결과를 공유하며 분위기를 부드럽게 만드는 용도로는 충분히 활용할 수 있습니다.

애플리케이션에서 제공되는 조언과 “사용 설명서”는 사용자의 응답을 바탕으로 생성형 AI가 작성한 텍스트입니다. 다만 일본어 문장 안에 “Respondent”라는 단어가 어색하게 등장하는 것을 볼 수 있습니다. 이는 생성형 AI 기반 애플리케이션에서 흔히 나타나는 문제를 보여줍니다.

표현력은 높아지지만, 품질 리스크도 함께 증가합니다.

또한 이 프로젝트는 취미 개발이기 때문에, 품질 보장을 위한 비용과 복잡성은 최소화하고 싶었습니다. 그래서 다음과 같은 원칙을 채택했습니다.

채점 로직은 deterministic하게 유지하고, 생성형 AI는 보조적인 텍스트 생성에만 사용한다.

이 접근을 통해 할루시네이션의 영향을 제한하면서도 AI의 표현력을 활용할 수 있습니다.

이제부터 실제로 겪었던 문제와 트레이드오프를 세 가지 관점에서 설명하겠습니다.

## 1. 어뷰즈 대응

이 애플리케이션은 익명으로 접근할 수 있지만, 백엔드에서는 유료 서비스인 Amazon Bedrock을 사용하고 있습니다. 따라서 봇에 의한 남용은 곧 비용 증가로 이어질 수 있습니다.

그래서 과도한 대응보다는 다음과 같은 방침을 선택했습니다:

리스크가 낮은 공격은 수용하고, 비용에 영향을 줄 수 있는 공격만 차단한다.

### 1.1 WAF를 통한 일반 공격 트래픽 차단

WAF 설정은 AWS의 기본 보호 패키지를 기반으로 다음을 적용했습니다:

- Geo 기반 차단 규칙  
- AWS 관리형 IP 평판 리스트  
- 익명 IP 차단 규칙  
- 애플리케이션에 맞춘 Rate 기반 제한  

한 가지 고민이 있었던 부분은 Lambda Function URL을 직접 공개하면 CloudFront를 거치지 않는 요청이 WAF를 우회할 수 있다는 점입니다. 이를 방지하기 위해 IAM 인증이나 커스텀 헤더 검증을 사용할 수 있지만, 환경 의존적인 구현과 운영 복잡성이 증가합니다.

리스크 대비 비용을 고려해 이번에는 해당 대응을 보류했습니다.

덧붙여, AWS 콘솔에서 직접 설정한 값은 잊어버리기 쉽습니다. 이를 방지하기 위해 설정은 코드로 관리하는 것을 추천합니다.

예를 들어 WAF의 경우:

1. JSON을 Source of Truth로 관리  
2. 변경은 항상 Source를 통해 반영  
3. CI에서 실제 설정과의 정합성을 검증  

흥미롭게도, 이 애플리케이션 규모에서는 WAF의 고정 비용이 Bedrock 사용량보다 더 높았습니다. 사전 비용 산정이 중요하다는 교훈을 얻었습니다 😅

### 1.2 단순 봇 및 직접 호출 방지

간단한 봇을 방지하기 위해 다음과 같은 방식을 구현했습니다:

1. 시각 정보와 시크릿을 기반으로 토큰 생성 후 쿠키로 전달  
2. 브라우저는 credentials: 'include'로 쿠키를 전송  
3. 서버는 유효 시간 범위를 벗어난 요청을 차단  

추가로:

- SameSite=Strict로 크로스 사이트 쿠키 전송 차단  
- CORS 허용 오리진 제한  

고급 봇은 우회 가능하지만, 현재 단계에서는 충분히 실용적인 대응입니다.

## 2. 기술 선택

Lambda는 자연스러운 선택이었지만 다음 두 가지를 검토했습니다:

1. 컨테이너 방식 (PHP)  
2. Zip 방식 (Python/Node)  

초기에는 기존 PHP 환경을 활용할 수 있는 컨테이너 방식이 매력적이었지만, AWS와의 통합성과 CI 복잡성을 고려해 Zip 방식을 선택했습니다.

결과적으로 AWS 환경 이해에 더 많은 시간이 들었기 때문에 적절한 선택이었다고 판단합니다.

또한 개발 효율을 위해:

- Lambda와 로컬을 공통으로 처리하는 템플릿  
- 테스트용 런처  

를 준비했고, 이는 이후 프롬프트 튜닝에서 큰 도움이 되었습니다.

API 구성은 다음 두 가지를 비교했습니다:

1. CloudFront + WAF → Lambda → Bedrock  
2. CloudFront + WAF → API Gateway → Lambda → Bedrock  

현재 요구사항에서는 API Gateway의 비용 대비 이점이 적어 도입하지 않았습니다.

## 3. 애플리케이션 개발

이제 본격적인 개발 단계입니다.

Reddit 등 글로벌 사용자에게 제공하기 위해 9개 언어를 지원하는 간단한 i18n 시스템을 구현했습니다.

```
const GREETING = i18n.text({
    en : 'Hello',
    ja : 'こんにちは',
    fr : 'Bonjour',
    de : 'Hallo',
    es : 'Hola',
    pt : 'Olá',
    hi : 'नमस्ते',
    ko : '안녕하세요',
    zh : '你好'
})
```

또한 LLM 출력을 안정화하기 위해 다음과 같은 설계를 적용했습니다:

- 사용자 인터페이스는 각자의 모국어  
- LLM 입력은 영어로 통일  

<img src='https://raw.githubusercontent.com/nakayama-kazuki/202x/main/pj-corridor.net/personalitytest/blog/arch.png' width='600' />

이렇게 하면 프롬프트 안정성과 다국어 대응을 동시에 확보할 수 있습니다.

마지막으로 프롬프트 튜닝입니다.

효율적인 반복 실험을 위해:

1. 테스트용 단축 기능 구현  
2. LLM 없이 프롬프트를 콘솔에 출력  
3. ChatGPT나 Gemini로 결과 평가  
4. 반복적으로 개선  

을 수행했습니다.

LLM의 피드백은 프롬프트를 비대하게 만드는 경향이 있기 때문에, 주기적으로 정리하고 구조를 리팩터링하는 것이 중요합니다.

마지막으로 “Respondent” 문제입니다. 동료용 문장에서는 “응답자”라는 명사가 필요했지만, 다음과 같은 지시는:

```
consistently use the {{lang}} term for "Respondent"
```

결과적으로 “당신”으로 출력되었습니다. 대명사를 금지하면 이번에는 “Respondent”가 그대로 출력됩니다. 이는 LLM이 자연스러운 표현을 우선하기 때문입니다.

여러 차례 튜닝을 시도했지만 안정적으로 특정 단어를 강제하는 것은 어려웠고, 최종적으로 다음과 같은 방법을 선택했습니다:

```
always use the fixed keyword "_RESPONDENT_" without any modification, translation, or suffixes.
```

그리고 클라이언트에서 치환합니다:

```
const RESPONDENT = i18n.text({
    en : 'Respondent',
    ja : '回答者',
    fr : 'Répondant',
    de : 'Befragter',
    es : 'Encuestado',
    pt : 'Respondente',
    hi : 'उत्तरदाता',
    ko : '응답자',
    zh : '受访者'
});
```

다소 단순한 방법처럼 보이지만 😅, 중요한 교훈이 있습니다:

자연어로 표현을 통제하려 하기보다는, 특정 키워드 형태로 출력한 뒤 후처리하는 방식이 더 안정적인 경우도 있습니다.
