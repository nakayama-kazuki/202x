Keep the scoring logic deterministic, and use the LLM only for supplementary text.

## 1. Abuse Prevention

The application allows anonymous access, but the backend relies on a paid AI service (Amazon Bedrock). This means abuse—especially automated requests—could directly lead to unexpected costs.

Rather than over-engineering defenses, I adopted a pragmatic policy:

Accept low-risk threats, and block high-impact ones.

### 1.1 Blocking Common Attack Traffic with WAF

For WAF, I based the configuration on AWS’s standard protection packs and selected:

- Geo-based rules (blocking regions with high attack traffic)
- AWS-managed IP reputation lists
- Anonymous IP detection rules
- Custom rate-based rules for this application

One concern was that when using Lambda Function URLs as a public endpoint, requests that do not go through CloudFront can bypass WAF. While it is possible to enforce CloudFront usage (for example, by using IAM authentication with OAC or validating custom headers), these approaches introduce additional operational complexity.

Considering the likelihood and impact of abuse, I decided not to implement these measures at this stage.

As a side note, configurations made directly in the AWS console are easy to forget. To avoid issues caused by temporary settings left unchanged, I recommend managing such configurations in your repository.

For example, with WAF:

1. Store the protection pack JSON as the source of truth  
2. Apply changes through the source of truth before updating AWS  
3. Validate consistency between the repository and deployed configuration in CI  

Interestingly, for this application’s traffic scale, the fixed cost of WAF exceeded the usage cost of Bedrock. This was a valuable lesson—always estimate costs beforehand 😅

### 1.2 Controlling Access from Simple Bots

To mitigate basic bot activity and direct API access, I implemented:

1. A token generated from a secret and timestamp, sent via Set-Cookie  
2. The browser includes the token using credentials: 'include'  
3. The server validates the token within an acceptable time window  

Additional measures:

- SameSite=Strict to prevent cross-site cookie usage  
- Restrict Access-Control-Allow-Origin to a whitelist  

These measures can be bypassed by sophisticated bots, but they are sufficient as an initial layer aligned with the project’s scope.

## 2. Technology Choices

Lambda was a natural choice for this application, but I evaluated two approaches:

1. Custom container (PHP)  
2. Zip deployment (Python/Node)  

Initially, the container approach seemed attractive due to existing local PHP environments. However, considering AWS integration and CI complexity, I chose the zip-based approach.

In hindsight, most of the effort went into understanding AWS behavior rather than application logic, so this decision proved appropriate.

To support development and testing, I also prepared:

- A shared Python template for Lambda and local environments  
- A local test launcher  

These tools later became essential for prompt tuning.

For API architecture, I considered:

1. CloudFront + WAF → Lambda → Bedrock  
2. CloudFront + WAF → API Gateway → Lambda → Bedrock  

Given that authentication, routing, and rate limiting were already handled elsewhere, I decided not to introduce API Gateway due to cost considerations.

## 3. Application Development

With infrastructure decisions made, development could begin.

Since I wanted users worldwide (e.g., from Reddit) to try the app, I implemented a lightweight i18n system supporting nine languages.

```
const GREETING = i18n.text({
    en : 'Hello',
    ja : 'こんにちは',
    fr : 'Bonjour',
    de : 'Hallo',
    es : 'Hola',
    pt : 'Olá',
    hi : 'नमस्ते',
    ko : '안녕하세요',
    zh : '你好'
})
```

To stabilize LLM outputs, I designed the interface so that user interactions remain in the native language, while input to the LLM is standardized in English.

Shared logic between CAPS and DiSC was extracted into reusable components to support future extensions (e.g., MBTI).

Finally, I focused on improving output quality and stability through prompt engineering. Rather than rushing into tuning, I first built a foundation for iterative experimentation:

1. Prepare shortcut features for prompt generation  
   - For example, automate “random answers + diagnostic query execution”  
   - Protect the shortcut using a hashed secret to prevent abuse  
2. Tune prompts in a test environment (without LLM calls)  
   1. Execute the shortcut  
   2. Output the generated prompt to the console  
   3. Evaluate it using tools like ChatGPT or Gemini  
   4. Refine the template and repeat until results are acceptable  
3. Final tuning in production (with LLM calls enabled)  
   1. Use shortcuts to detect regressions  
   2. Run full flows and validate output quality  

In practice, LLM suggestions tend to make prompts overly verbose, so periodically consolidating instructions and refactoring structure is important.

Returning to the earlier issue with “Respondent”: in the colleague-facing text, the grammatical subject needed to be expressed as “the respondent.” However, the following instruction:

```
The second response must be written for the Respondent's colleagues or friends.
The grammatical subject must be the colleagues or friends, and when referring to the Respondent,
consistently use the {{lang}} term for "Respondent".
```

resulted in the subject being rendered as “you.” When adding a constraint to prohibit pronouns, the output instead reverted to “Respondent.” This happens because LLMs tend to prioritize natural expressions over strict adherence to instructions.

Despite repeated tuning, it proved difficult to consistently enforce “回答者” (or its equivalent) purely through natural language constraints. The final solution was:

```
The second response must be written for the Respondent's colleagues or friends.
The grammatical subject must be the colleagues or friends, and when referring to the Respondent,
always use the fixed keyword "_RESPONDENT_" without any modification, translation, or suffixes.
```

In this approach, the keyword \_RESPONDENT\_ is emitted as-is and then replaced on the client side:

```
const RESPONDENT = i18n.text({
    en : 'Respondent',
    ja : '回答者',
    fr : 'Répondant',
    de : 'Befragter',
    es : 'Encuestado',
    pt : 'Respondente',
    hi : 'उत्तरदाता',
    ko : '응답자',
    zh : '受访者'
});
```

This is admittedly a workaround 😅. However, it highlights an important insight: instead of trying to control wording through natural language instructions, it is often more reliable to emit fixed tokens and handle them in post-processing.
