## 1. Abuse Prevention

The assessment app is publicly accessible without authentication, but the backend uses a paid generative AI service (Amazon Bedrock). This makes “getting hit by bot abuse and blowing up costs” a very real scenario.

Taking into account the attacker’s incentives, I adopted the following policy:

**Accept low-impact threats and block everything else.**

Based on this, I implemented the following measures.

---

### 1.1 Blocking Common Attack Traffic with WAF

For WAF, I started with AWS-managed rule sets and customized them:

- `GeoRule` (block IPs from high-risk regions)
- `AWSManagedRulesAmazonIpReputationList` (known malicious IPs)
- `AWSManagedRulesAnonymousIpList` (anonymous / proxy IPs)
- Tuned `GlobalRateBasedRule` (request limits)
- Tuned `RateBasedRulePOST` (limits on POST / PUT / DELETE)

I removed rules that were unnecessary for this application.

One concern is that when using Lambda Function URLs as a public endpoint, requests that bypass CloudFront can also bypass WAF.

There are ways to enforce routing through CloudFront, but they require additional implementation:

- Switching `AuthType` to `AWS_IAM` and using CloudFront OAC  
  👉 Requires handling `x-amz-content-sha256` for POST requests  
- Adding custom headers in CloudFront and validating them in Lambda  
  👉 Requires managing and validating secret values  

Considering the risk of someone discovering the Lambda URL and bypassing WAF, I decided not to implement these measures for now.

As a side note, settings configured in the AWS console are easy to forget. Leaving temporary settings in place can cause issues later. To avoid this, I recommend managing configurations in a repository.

For example, with WAF:

1. Store the JSON as the Source of Truth
2. Always update the Source of Truth first, then apply it in the console
3. Verify consistency via CI

One more thing—an unexpected outcome:

At my traffic scale, the fixed cost of WAF exceeded the usage-based cost of Bedrock 😅

A good reminder: always estimate costs upfront.

For now, I’m considering disabling WAF and instead implementing request limits using DynamoDB until traffic grows.

---

### 1.2 Controlling Access from Simple Bots

For additional protection, I implemented:

1. Generate a token from a secret + timestamp and send it via `Set-Cookie`
2. The browser includes it in requests (`credentials: 'include'`)
3. The server validates whether the token timing matches realistic human behavior

Requests outside the expected time range are treated as bots or abandoned sessions.

Additional measures:

- `SameSite=Strict` to block cross-domain POST requests
- Restrict `Access-Control-Allow-Origin` to a whitelist

These measures can be bypassed by sophisticated bots, but they are sufficient as an initial layer of defense.

## 2. Technology Choices

Next, I’ll describe my decisions around Lambda and API Gateway.

Lambda was a natural choice, but I considered:

1. Container-based (PHP)
2. Zip-based (Python / Node.js)

Initially, the container approach was appealing because I already had a local PHP environment. However, considering Lambda compatibility and CI complexity, I chose the zip-based approach.

In hindsight, I spent more time experimenting with AWS than writing application code, so this was the right decision.

Also, repeatedly creating Lambda functions from the console generates IAM roles each time. These can quickly become technical debt, so it’s best to clean them up regularly.

Before moving on to application development, I resisted the urge to rush ahead and instead prepared a foundation to make things easier later:

- Shared Python template for local and Lambda environments
- Local test launcher

These became the basis for prompt tuning later.

---

### API Gateway

I considered the following architectures:

1. CloudFront + WAF → Lambda → Bedrock  
2. CloudFront + WAF → API Gateway → Lambda → Bedrock  

However, given the current implementation:

| Feature | Implementation |
|--------|----------------|
| Auth | Custom access control |
| Routing | Routing fully handled within Lambda |
| Throttling | Handled by WAF (subject to change) |

I concluded that introducing API Gateway at this stage would not provide enough benefit to justify the cost.

### Prompt Tuning

Now for the final step: improving and stabilizing AI output.

Instead of rushing, I followed a “slow is smooth, smooth is fast” approach and built a proper foundation:

1. Shortcut for generating prompts  
2. Tuning in a test environment (no AI)
   - Output prompts
   - Evaluate with ChatGPT / Gemini
   - Iterate
3. Tuning in production
   - Adjust based on output differences
   - Validate end-to-end behavior

One issue is that incorporating feedback from LLMs tends to make prompts grow over time. Periodic refactoring and consolidation are important.
